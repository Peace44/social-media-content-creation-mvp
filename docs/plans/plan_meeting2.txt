 Plan: Fix the 3 Competitor-Analysis Bugs (Meeting 2)                                                                                                                                                                                                                                                                                                                                                                                                                                                                            Context                                                                                                                                                                                                                                                        

 Client Sid (Kolif Agency, IG: @sid_consulentemarketing) reported three defects after testing the MVP:

 1. Wrong competitor identification — Claude returns generic SME marketing consultants (Bertuol, Severgnini, Ingala, Marabini) instead of the true niche (marketing per coach e counselor). True competitors like Federico Ferrarelli, Ivano Peggion, and       
 Francesco Conte are never surfaced.
 2. "Social profiles" UI field renders empty even when follower counts are populated.
 3. No persistence — closing the app / losing connection wipes the entire analysis; no history exists.

 The goal is to fully resolve all three, tackled from simplest to hardest so each fix is shippable and testable before moving on.

 Fix 2 — Bug 3: Persistence & history (medium)

 Files to create/modify:
 - NEW: src/competitor_analysis/storage/history.py
 - app.py — add a sidebar "History" section and wire saves into the submit flow
 - src/competitor_analysis/models.py — add AnalysisRecord wrapper model

 Design (JSON-per-analysis, reusing the .cache/search/ pattern from search.py:15-49):
 1. Storage dir: <repo>/.cache/history/ — one JSON file per analysis, filename {timestamp}_{url_slug}.json.
 2. AnalysisRecord Pydantic model containing: id, created_at, input_url, profile: ProfileSummary, rows: list[CompetitorRow]. Serialize via .model_dump(), deserialize via .model_validate().
 3. history.py exposes: save_analysis(record) -> str, load_analysis(id) -> AnalysisRecord, list_analyses() -> list[AnalysisRecordMeta], delete_analysis(id) -> None.
 4. app.py changes:
   - After analyze(...) succeeds, call save_analysis(...); keep st.session_state["current_analysis_id"] so reruns don't re-save.
   - Sidebar: st.sidebar.title("Cronologia") → list recent analyses (newest first), clicking loads the record into st.session_state["loaded_record"] and re-renders the cards without calling any Claude API.
   - Add a "🗑" button per row to delete.
   - On submitted, prefer the freshly-computed rows; otherwise render loaded_record.rows if present; otherwise the empty-state message at app.py:260-261.

 Reused utilities:
 - _save_cache / _load_cache / _cache_path pattern from src/competitor_analysis/scraper/search.py:20-49 — mirror the structure (MD5 not needed here since URL slug + timestamp is fine).
 - _flatten / export_excel from src/competitor_analysis/output/export.py already work with CompetitorRow[] — "Export Excel" button can operate on loaded history records unchanged.

 Why this is second: touches more files than Fix 1 but is a pure additive feature — no changes to the analysis pipeline or prompts.

 Verification:
 - Run an analysis → confirm a JSON file appears in .cache/history/.
 - Hard-reload the Streamlit page → sidebar shows the prior analysis; clicking it re-renders all cards without any Claude API spend (watch logs / _call_claude stays silent).
 - Kill streamlit run mid-session, restart → history persists.
 - Delete button removes the file and the sidebar entry.

 Fix 3 — Bug 1: Correct competitor identification (hardest)

 Files to modify:
 - src/competitor_analysis/scraper/profile.py — add headless-browser fallback
 - src/competitor_analysis/analysis/competitor_finder.py — retune queries, add post-filter
 - requirements.txt / pyproject.toml — add playwright (+ playwright install chromium note)

 Changes:

 A. Scraping upgrade (profile.py):
 1. Keep the fast httpx path as the primary.
 2. On empty / login-wall response (detect "Accedi" / "Log in" / empty og:description), fall back to a Playwright headless-chromium fetch with Italian locale, realistic viewport, and explicit wait for meta[property="og:description"]. Instagram's
 og:description meta tag exposes the bio + follower count without login for public profiles.
 3. Also extract the handle's public JSON surface when available (window._sharedData or <script type="application/ld+json">).
 4. Extend RawProfile (models.py) with a bio: str field populated from og:description — this is the highest-signal text Claude will see.

 B. Search-query retuning (competitor_finder.py:165-184):
 1. Replace generic templates (f"agenzia marketing {audience_short} {geo}") with niche-specific ones that preserve micro-niche tokens unchanged:
   - f'"{profile.niche}" Instagram Italia'
   - f"marketing per {profile.target_audience} Italia"
   - f"consulente social media {profile.target_audience}"
   - f'"coach" OR "counselor" marketing consulente Italia' (only when niche contains "coach"/"counselor" tokens)
 2. Add a token-extraction helper that pulls 2-3 salient micro-niche words from profile.niche (e.g. "coach", "counselor") and mandates they appear in at least one query.

 C. Candidate post-filter / re-rank (new function in competitor_finder.py, called from find_competitors after the existing Claude call, before return):
 1. For each candidate, issue a lightweight per-candidate verification Claude call: given the client's niche + audience + the candidate's description/primary_url, return { "is_direct_competitor": bool, "confidence": 0-1, "reason": str }.
 2. Drop any candidate with is_direct_competitor == false or confidence < 0.6.
 3. Sort survivors by confidence descending.
 4. Cache verifications in .cache/verify/ using the same pattern so re-runs are cheap.
